# Judgment Tamagotchi Loop

Status: proposed  
Scope: product loop, progression meaning, autoresearch target shape  
Depends on: `docs/product-specs/core.md`, `docs/product-specs/raising.md`, `docs/AGENT_SYSTEM_DESIGN.md`, `docs/BATTLEFIELD_DESIGN.md`

## Purpose

This document defines what makes Cogochi meaningful as an agent-training platform instead of:

- a sterile prompt lab
- a cute but shallow pet sim
- a chart tool wearing game art

The intended product shape is:

`judgment tamagotchi -> authored doctrine -> visible agent change -> controllable chart test -> reflection -> next care action`

## Product Thesis

Cogochi should feel like the player is raising a personal AI chart companion whose:

- judgment style is authored by the player
- visible form changes as training accumulates
- battle behavior proves whether that training actually worked
- history leaves keepsakes, scars, and specialized identity behind

If the player cannot answer both of these after a session, the loop is not meaningful yet:

1. `How did my agent change?`
2. `Did that change improve judgment on the chart?`

## Target Aesthetic Mix

Primary:

- `Fantasy`: this little agent is mine
- `Challenge`: the chart still demands correct judgment

Secondary:

- `Expression`: my doctrine and habits shape the agent
- `Discovery`: repeated runs reveal hidden strengths, weaknesses, and memory patterns

Avoid optimizing primarily for:

- `Submission`: idle background pet care
- `Sensation`: spectacle that does not improve judgment readability

## Meaningful Platform Test

The platform becomes meaningful only when all five are true:

### 1. Authored Self

The player can inject:

- indicator presets
- script rules
- doctrine notes
- memory bias and curated notes
- risk style

The agent must feel authored, not picked from a preset box.

### 2. Visible Body Change

Training must not only change hidden numbers.
It must also change the companion through:

- silhouette additions
- accessory upgrades
- stance changes
- aura or memory marks
- keepsakes from important matches

### 3. Deterministic Proof

The player must be able to run the same historical scenario and compare:

- before
- after
- what changed
- why the result changed

### 4. Ongoing Care

The player needs a reason to come back that is more meaningful than feeding a mascot.

Cogochi care should be:

- pruning stale memory
- rewriting weak doctrine
- calming a shaken agent after bad reads
- sharpening a specialist before a fitting scenario
- restoring squad cohesion

### 5. Persistent Identity

The roster must remember what happened.
Agents need durable history such as:

- lesson ribbons
- battle marks
- trust/bond increase
- specialization tags
- signature doctrine traits

## Loop Layers

### Micro Loop: 1 to 3 minutes

1. Inspect one agent
2. Perform one care or training action
3. See one visible delta preview
4. Enter one controllable chart battle
5. Read one reflection
6. Queue one next improvement

### Core Loop: 10 to 20 minutes

1. Tune one agent around a market regime or doctrine idea
2. Run several fixed scenarios
3. Compare outcomes
4. Lock in a specialization
5. Earn one visible form change or keepsake

### Meta Loop: days to weeks

1. Build a stable of personal agents
2. Maintain different specialists for different chart conditions
3. Grow a personal field manual through memory and doctrine
4. Assemble squads whose identity reflects the player's own pattern library

## Growth Axes

Every growth axis should have a gameplay meaning and a visual meaning.

| Axis | Gameplay meaning | Visual meaning |
| --- | --- | --- |
| Doctrine clarity | Better reasoning and call quality | notebook charm, antenna glyph, halo line |
| Memory quality | Better retrieval and trap recall | memory sigil, satchel, archive badge |
| Risk discipline | Better vetoes and survivability | shield mark, grounded stance, guard shell |
| Execution tempo | Faster commitment and cleaner push | forward lean, trail mark, blade or thruster cue |
| Bond | Higher trust and steadier expression | friendlier idle, emotes, paired keepsakes |

## Transformation Model

Use a layered transformation system instead of random monster evolution.

### Base Layer

- species silhouette
- role identity

### Growth Layer

- doctrine accessory
- memory accessory
- risk/execution stance

### History Layer

- keepsakes from memorable matches
- ribbons for milestones
- field marks from failures survived

### Signature Layer

Unlocked only after repeated validated behavior.

Examples:

- a breakout specialist gains a persistent forward motion cue
- a trap reader gains a visible alert crest
- a disciplined defender gains heavier shield language

## Care Loop Without Chores

Do not use hunger, sleep, or bathroom meters.

Use judgment-native care states instead:

- `memory drift`: useful lessons are buried by noise
- `doctrine blur`: the rule set became vague or contradictory
- `confidence shake`: recent failure reduced commitment quality
- `squad friction`: role fit and trust weakened after bad coordination

Matching care actions:

- `curate`
- `annotate`
- `spar`
- `restabilize`
- `retune`

## Battle Proof Rules

A visible character change is only valid if it connects to battle proof.

Each accepted training change should affect at least one of:

- command deck quality
- reveal quality
- veto timing
- commitment timing
- reflection clarity

If the body changes but battle behavior does not, it is cosmetic drift.
If battle behavior changes but the body never reflects growth, attachment remains weak.

## Autoresearch Implications

The autoresearch loop should not ask only:

- `did win rate improve?`

It must also ask:

- did ownership increase?
- did visible transformation become clearer?
- did the same scenario show a meaningful before/after change?
- did the player receive one strong reason to come back?

## Non-Goals

- tamagotchi chores detached from chart logic
- random evolution trees with no judgment meaning
- cosmetic-only skins sold as progression
- agent growth that cannot be explained in battle terms
- battle spectacle that hides training causality

## Design Acceptance Questions

The loop is on target only if the answer to these questions trends toward `yes`:

1. Can the player describe one agent as `mine` instead of `a preset`?
2. Can the player point to a visible companion change after training?
3. Can the player rerun a fixed scenario and explain the delta?
4. Does a bad run create a new care action instead of only a failure screen?
5. Does the roster look more like a traveled squad over time than a static database?
